# 需求详情：OpenClaw 工作区能力基线

## 1. 主题目标与价值说明
本主题用于定义“每个 agent 工作区都应具备哪些 OpenClaw 风格基础能力”。目标不是复制某个单一目录的文件摆放，而是沉淀一套可跨工作区复用的统一能力基线，使每个工作区都具备：

1. 稳定的身份与上下文恢复能力。
2. 每日记忆与长期记忆能力。
3. 心跳式维护与主动整理能力。
4. 可扩展的本地技能与工具说明入口。
5. 清晰的安全边界与读取规则。

## 2. 用户画像与使用场景
### 用户画像
1. Owner：同时维护多个 agent 工作区，希望每个工作区都能“开箱即恢复记忆”。
2. 单个工作区内的 agent：每次重启后都需要快速知道“我是谁、我在帮谁、我最近做了什么”。
3. 执行改造的维护者：需要知道哪些能力是必须补齐的，哪些只是可选增强。

### 使用场景
1. 直接主会话：agent 启动后加载身份、用户上下文、近两天短期记忆和长期记忆。
2. 非主会话/共享会话：agent 启动后只加载安全可共享的上下文，不加载长期个人记忆。
3. 日常工作中：agent 将重要事实写入每日记忆，并定期提炼到长期记忆。
4. 心跳巡检时：agent 读取心跳清单、执行轻量巡检、更新记忆或保持静默。

## 3. 用户旅程或关键流程
1. 工作区初始化：
   - 创建或补齐 `AGENTS.md` 与 `.codex/` 标准结构。
2. 会话启动：
   - 读取 `.codex/SOUL.md`。
   - 读取 `.codex/USER.md`。
   - 读取 `.codex/memory/YYYY-MM-DD.md`（今天与昨天）。
   - 若为主会话，再读取 `.codex/MEMORY.md`。
3. 会话进行中：
   - 将重要决策、上下文、约束写入每日记忆。
   - 必要时读取 `.codex/TOOLS.md`、`.codex/HEARTBEAT.md`、`.codex/skills/`。
4. 周期维护：
   - 从每日记忆提炼长期记忆。
   - 清理过时记忆并保留可追溯记录。

## 4. 功能需求清单
### FR-OCB-01 工作区必须保留顶层 `AGENTS.md`
1. 每个 agent 工作区顶层必须存在 `AGENTS.md`，作为唯一强制保留在顶层的 agent 工作文档入口。
2. `AGENTS.md` 必须承担启动读取顺序、记忆规则、安全边界和工具入口说明。

### FR-OCB-02 身份与上下文文档必须收纳到 `.codex/`
1. 每个 agent 工作区必须支持以下核心文档位于 `.codex/` 下：
   - `.codex/SOUL.md`
   - `.codex/USER.md`
   - `.codex/MEMORY.md`
   - `.codex/TOOLS.md`
   - `.codex/HEARTBEAT.md`
2. 以下文档为条件性文档，若工作区需要对应能力，则也必须放在 `.codex/` 下：
   - `.codex/IDENTITY.md`
   - `.codex/BOOTSTRAP.md`
3. 不得再依赖顶层同名文件作为正式读取入口。

### FR-OCB-03 统一会话启动读取顺序
1. `AGENTS.md` 必须定义统一读取顺序。
2. 默认顺序至少应包含：
   - `.codex/SOUL.md`
   - `.codex/USER.md`
   - `.codex/memory/YYYY-MM-DD.md`（今天）
   - `.codex/memory/YYYY-MM-DD.md`（昨天）
3. 主会话必须额外读取 `.codex/MEMORY.md`。
4. 非主会话默认不得读取 `.codex/MEMORY.md`，除非存在更高优先级的安全授权规则。

### FR-OCB-04 每日记忆机制
1. 每个 agent 工作区必须支持 `.codex/memory/YYYY-MM-DD.md` 形式的每日记忆。
2. 当日文件不存在时，工作区必须支持在首次写入时自动创建。
3. 每日记忆应记录：
   - 重要决策；
   - 近期任务上下文；
   - 待跟进事项；
   - 对后续会话有恢复价值的事实。

### FR-OCB-05 长期记忆机制
1. 每个 agent 工作区必须支持 `.codex/MEMORY.md` 作为长期记忆主档。
2. 长期记忆应承载跨天、跨回合仍有价值的稳定信息，而非流水日志。
3. 工作区必须定义“从每日记忆提炼到长期记忆”的维护规则。

### FR-OCB-06 心跳与主动维护能力
1. 每个 agent 工作区必须支持 `.codex/HEARTBEAT.md` 作为心跳巡检清单。
2. 若工作区启用了心跳轮询，必须支持记录最近检查状态，例如 `.codex/memory/heartbeat-state.json`。
3. 心跳能力至少应支持：
   - 读取待办检查项；
   - 在无新信息时保持静默；
   - 在需要时更新每日记忆或长期记忆。

### FR-OCB-07 工具说明与技能入口
1. 每个 agent 工作区必须支持 `.codex/TOOLS.md` 作为本地工具、环境约束和操作惯例说明。
2. 若工作区支持本地技能，则本地技能统一放在 `.codex/skills/`。
3. `AGENTS.md` 或相关导引文档必须能将 agent 引导到 `.codex/skills/`，而不是依赖顶层 `skills/` 作为默认入口。

### FR-OCB-08 安全边界与共享上下文隔离
1. 工作区必须区分“可在共享会话中读取的信息”和“仅主会话可读取的信息”。
2. `.codex/MEMORY.md` 默认按长期私人记忆处理，只在主会话加载。
3. 对外发送、外部平台操作、破坏性命令等高风险行为，仍必须由各工作区在 `AGENTS.md` 中定义独立门禁。

### FR-OCB-09 写入优先级规则
1. 工作区必须明确“想记住就写文件，不依赖临时上下文”的规则。
2. 用户明确说“记住这个”时，agent 必须优先写入每日记忆或相关长期档，而不是仅在对话中口头确认。

### FR-OCB-10 缺省补齐能力
1. 当前尚无 `.codex/` 或记忆链路的工作区，改造时必须至少补齐最小可运行能力：
   - 顶层 `AGENTS.md`
   - `.codex/SOUL.md`
   - `.codex/USER.md`
   - `.codex/MEMORY.md`
   - `.codex/TOOLS.md`
   - `.codex/HEARTBEAT.md`
   - `.codex/memory/`
2. 缺省补齐后，工作区才算具备“有记忆能力”的最低门槛。

## 5. 非功能需求
1. 一致性：所有 agent 工作区的读取顺序、文档命名和记忆主路径必须统一。
2. 安全性：长期记忆不得在共享会话中默认泄露。
3. 可维护性：新增工作区时应能直接复用同一模板，减少人工补配置。
4. 可追溯性：记忆写入与提炼过程必须能通过文件追溯，而不是依赖临时对话状态。
5. 低噪音：顶层不再堆放 agent 内部工作文档，降低浏览和切换成本。

## 6. 验收标准
### AC-OCB-01 主会话长期记忆加载
1. Given 某目录已完成 OpenClaw 风格能力改造
2. When 在主会话中启动该工作区
3. Then agent 会按 `AGENTS.md` 读取 `.codex/SOUL.md`、`.codex/USER.md`、近两天 `.codex/memory/*.md`，并额外读取 `.codex/MEMORY.md`

### AC-OCB-02 非主会话长期记忆隔离
1. Given 某目录已完成 OpenClaw 风格能力改造
2. When 在共享/非主会话中启动该工作区
3. Then agent 默认不读取 `.codex/MEMORY.md`

### AC-OCB-03 每日记忆可写
1. Given 某工作区当天尚无 `.codex/memory/YYYY-MM-DD.md`
2. When agent 首次记录当天关键事实
3. Then 系统会创建当天记忆文件并写入内容

### AC-OCB-04 技能入口统一
1. Given 某工作区支持本地技能
2. When agent 需要读取本地技能
3. Then 默认入口应为 `.codex/skills/`，而不是旧的顶层 `skills/`

### AC-OCB-05 心跳清单可读取
1. Given 某工作区启用了心跳巡检
2. When 心跳触发
3. Then agent 能读取 `.codex/HEARTBEAT.md` 并执行其中的轻量巡检或静默策略

### AC-OCB-06 验收证据
1. Given 某工作区完成改造
2. When 进行交付验收
3. Then 至少提供 1 组启动读取证据和 1 组记忆写入证据

## 7. 边界条件与异常处理
1. 对于不属于主会话的上下文，即使存在 `.codex/MEMORY.md`，也不得默认加载。
2. 对于尚未创建任何记忆文件的新工作区，必须允许从空状态初始化，而不是要求先手工铺满模板内容。
3. 对于暂不需要 `IDENTITY.md` 或 `BOOTSTRAP.md` 的工作区，可以不创建，但若存在，对应路径必须在 `.codex/` 下。
4. 若某工作区因业务安全要求不能启用心跳巡检，应保留 `.codex/HEARTBEAT.md` 的缺省约束入口，并在该文件中显式声明禁用策略。

## 8. 依赖项与开放问题
1. 依赖统一的 `.codex/` 模板与初始化脚本。
2. 依赖执行方具备对 `AGENTS.md` 和相关说明文档的批量更新能力。
3. 开放问题：是否要为 8 个 agent 工作区统一生成一套最小模板内容，还是允许每个工作区在统一路径下维持差异化正文。
