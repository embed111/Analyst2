# 需求详情：`.codex` 共享支撑目录治理

## 1. 主题目标与价值说明
本主题用于定义 `../.codex` 的目标状态。该目录不是独立 agent 工作区，而是跨工作区共享的隐藏支撑目录，当前已经承载：

1. 共享身份/记忆文档；
2. 共享脚本；
3. 共享技能。

本主题的核心目标不是把它改造成“顶层只保留 `AGENTS.md` 的普通工作区”，而是明确它作为共享支撑目录的例外规则，避免后续改造时错误地将其再次嵌套为 `.codex/.codex/`。

## 2. 用户画像与使用场景
1. Owner：需要一个可复用的跨工作区共享能力根目录。
2. 各工作区执行方：需要知道哪些内容可复用自 `../.codex`，哪些必须在各自工作区本地落地。
3. 维护者：需要避免将共享目录误当作普通工作区处理。

## 3. 用户旅程或关键流程
1. 识别 `../.codex` 的共享属性。
2. 明确它与各工作区本地 `.codex/` 的边界。
3. 保留其现有共享结构，不做重复嵌套。

## 4. 功能需求清单
### FR-CDX-01 目录角色定义
1. `../.codex` 必须被定义为“共享支撑目录”，而非普通 agent 工作区。
2. 不要求为 `../.codex` 新增顶层 `AGENTS.md`。

### FR-CDX-02 保持当前共享结构
1. `../.codex` 允许保留当前顶层共享文档：
   - `SOUL.md`
   - `USER.md`
   - `MEMORY.md`
   - `TOOLS.md`
   - `HEARTBEAT.md`
   - `IDENTITY.md`
2. 允许保留：
   - `memory/`
   - `scripts/`
   - `skills/`

### FR-CDX-03 禁止重复嵌套
1. 不得将 `../.codex` 再改造成 `../.codex/.codex/` 结构。
2. 不得要求其满足“顶层只保留 `AGENTS.md`”的普通工作区规则。

### FR-CDX-04 明确复用边界
1. `../.codex` 可提供共享模板、共享技能与共享脚本。
2. 各工作区若需本地记忆与本地技能，仍应在各自根目录下拥有自己的 `.codex/`。
3. 共享目录不得直接替代工作区本地 `.codex/`。

## 5. 非功能需求
1. 清晰性：必须明确这是例外目录。
2. 可复用性：共享脚本和共享技能可被多个工作区引用。
3. 可维护性：不得因为统一治理而破坏现有共享资产。

## 6. 验收标准
1. Given `../.codex` 纳入治理
2. When 完成目录目标状态判定
3. Then 它被标记为“共享支撑目录”，且不要求新增 `AGENTS.md` 或再次嵌套 `.codex/`

## 7. 边界条件与异常处理
1. 若后续用户要求将 `../.codex` 升级为独立工作区，需要单独立项，不沿用本轮默认规则。
2. 若共享目录中的某项能力要下放到工作区本地，必须保留来源说明和职责划分。

## 8. 依赖项与开放问题
1. 依赖各工作区改造文档显式引用本目录的“共享支撑”角色。
2. 开放问题：未来是否需要为 `../.codex` 增加共享模板索引文档；本轮先不强制。
