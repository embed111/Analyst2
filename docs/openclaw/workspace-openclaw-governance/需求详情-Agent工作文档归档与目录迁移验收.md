# 需求详情：Agent 工作文档归档与目录迁移验收

## 1. 主题目标与价值说明
本主题用于定义“如何把 agent 工作文档及相关工作资产从工作区顶层收纳到 `.codex/`，并保证迁移后仍可正常使用”。核心价值如下：

1. 让工作区顶层保持简洁，只暴露 `AGENTS.md` 这一统一入口。
2. 避免记忆、人格、工具、心跳等 agent 内部资料散落在顶层。
3. 保证迁移不仅改文件位置，还同步修复引用、读取顺序和验收链路。

## 2. 用户画像与使用场景
### 用户画像
1. Owner：希望 9 个目录整体被纳入统一治理，而不是逐个靠记忆修修补补。
2. 执行改造的维护者：需要一份明确的迁移映射和逐目录验收清单。
3. 后续新建工作区的维护者：需要直接复用这套目录契约，避免再次出现顶层文档蔓延。

### 使用场景
1. 对现有 9 个目录进行一次性盘点和迁移。
2. 把 `../openclaw` 这种已有较完整能力、但顶层过于暴露的工作区收口到新契约。
3. 把尚未具备 `.codex/` 结构的工作区补齐。
4. 在迁移完成后，对所有目录做统一验收与回归检查。

## 3. 用户旅程或关键流程
1. 目录盘点：
   - 识别 9 个一级目录的当前状态。
   - 区分已有 agent 工作区、待补齐工作区、共享支撑目录。
2. 迁移设计：
   - 生成“旧路径 -> 新路径”映射表。
   - 明确哪些顶层文件/目录属于 agent 工作资产。
3. 目录迁移：
   - 创建或补齐 `.codex/`。
   - 移动 agent 工作文档和相关资产。
   - 保留顶层 `AGENTS.md`。
4. 引用修复：
   - 修复 `AGENTS.md`、脚本、技能说明、治理文档中的旧路径。
5. 验收回归：
   - 验证顶层约束。
   - 验证启动与记忆链路。
   - 验证无旧路径残留。

## 4. 功能需求清单
### FR-MIG-01 9 个目录必须全部纳入治理清单
1. 本轮治理对象固定为 `../` 下 9 个一级目录：
   - `../.codex`
   - `../agent_gen`
   - `../Analyst`
   - `../Analyst2`
   - `../openclaw`
   - `../PortraitRpDemo`
   - `../skill_manager`
   - `../trainer`
   - `../workflow`
2. 每个目录都必须有明确的目标状态，不允许遗漏。

### FR-MIG-02 目录类型必须显式分类
1. 每个目录必须被标记为以下之一：
   - agent 工作区；
   - 共享支撑目录；
   - 过渡态目录。
2. 对于非 agent 工作区，仍必须说明其为何纳入治理、治理后保留什么角色。

### FR-MIG-03 顶层 agent 工作文档只保留 `AGENTS.md`
1. 对于 agent 工作区，顶层只允许保留 `AGENTS.md` 作为 agent 工作文档。
2. 以下顶层文件若存在且属于 agent 工作文档，必须迁入 `.codex/`：
   - `SOUL.md`
   - `MEMORY.md`
   - `IDENTITY.md`
   - `USER.md`
   - `TOOLS.md`
   - `HEARTBEAT.md`
   - `BOOTSTRAP.md`
3. 普通项目文档如 `README.md` 不在本条强制迁移范围内，除非其内容直接承担 agent 工作文档职责。

### FR-MIG-04 agent 工作资产默认归档到 `.codex/`
1. 若 `memory/` 属于 agent 的每日记忆目录，目标路径应为 `.codex/memory/`。
2. 若 `skills/` 属于 agent 本地技能目录，目标路径应为 `.codex/skills/`。
3. 若 `tasks/` 属于 agent 内部任务目录，目标路径应优先为 `.codex/tasks/`。
4. 任何 agent 内部使用、但不应暴露在顶层的辅助资产，都应优先归类到 `.codex/`。

### FR-MIG-05 路径迁移必须伴随引用修复
1. 迁移完成后，所有纳入范围的 `AGENTS.md` 必须引用新路径。
2. 相关脚本、技能文档、治理文档若引用旧路径，也必须同步修复。
3. 不允许出现“文件已移动，但启动说明仍指向旧路径”的状态。

### FR-MIG-06 迁移必须可审计
1. 执行方必须保留每个目录的迁移前后清单。
2. 每个迁移动作必须有“来源路径、目标路径、是否修复引用、验证结果”的记录。
3. 若有例外目录或例外文件，必须单独说明原因。

### FR-MIG-07 迁移必须支持 dry-run 与回滚说明
1. 正式迁移前，执行方必须能输出 dry-run 结果。
2. 正式迁移后，必须保留回滚映射或恢复说明。
3. 不得在没有留痕的情况下直接删除历史事实。

### FR-MIG-08 现有 `../openclaw` 必须作为重点样本处理
1. `../openclaw` 当前同时具备：
   - 顶层 `SOUL.md`、`MEMORY.md`、`USER.md`、`TOOLS.md`、`HEARTBEAT.md`、`BOOTSTRAP.md`
   - 顶层 `memory/`、`skills/`、`tasks/`
   - `.codex/skills/`
2. 其迁移结果必须成为“能力完整但目录收口”的标准样本，供其他工作区对齐。

### FR-MIG-09 缺少 `.codex/` 的工作区必须补齐
1. 当前尚无 `.codex/` 的 agent 工作区，必须在迁移中补齐 `.codex/`。
2. 补齐后才允许将顶层 agent 工作文档迁入其中。

### FR-MIG-10 新工作区创建默认继承该契约
1. 本轮完成后，后续新增 agent 工作区应默认采用同一目录契约。
2. 不应再创建“顶层散落 agent 工作文档”的新工作区。

## 5. 非功能需求
1. 幂等性：重复执行迁移检查时，不应导致重复移动或路径错乱。
2. 可审计性：每个目录的迁移结果和例外必须可追溯。
3. 安全性：迁移不得破坏长期记忆隔离规则。
4. 兼容性：迁移后仍应兼容 `.codex/skills/` 等既有约定。
5. 可复用性：迁移规则应能沉淀为后续新工作区模板，而不是一次性手工操作。

## 6. 验收标准
### AC-MIG-01 9 目录覆盖
1. Given 本轮改造范围为 `../` 下 9 个一级目录
2. When 执行方提交迁移结果
3. Then 必须能逐目录列出目标状态、迁移动作和验收结论

### AC-MIG-02 顶层约束通过
1. Given 某目录被判定为 agent 工作区
2. When 迁移完成
3. Then 顶层不再保留除 `AGENTS.md` 以外的 agent 工作文档

### AC-MIG-03 `.codex` 路径生效
1. Given 某目录存在 `SOUL.md`、`MEMORY.md` 等 agent 工作文档迁移需求
2. When 迁移完成
3. Then 对应文档应位于 `.codex/` 下，且 `AGENTS.md` 指向新路径

### AC-MIG-04 记忆目录收口
1. Given 某目录原有顶层 `memory/`
2. When 迁移完成
3. Then 每日记忆正式入口应为 `.codex/memory/`

### AC-MIG-05 无旧路径残留
1. Given 某目录已完成迁移
2. When 对纳入范围的 `AGENTS.md`、脚本、技能文档和治理文档进行搜索
3. Then 不应再出现旧的顶层 agent 工作文档读取路径

### AC-MIG-06 证据完整
1. Given 本轮目录迁移已完成
2. When 进行最终验收
3. Then 每个目录至少提供：
   - 迁移前后目录清单；
   - 路径修复证明；
   - 启动或读取验证结果；
   - 例外说明（如有）

## 7. 边界条件与异常处理
1. 若某目录当前没有 `AGENTS.md`，但被纳入本轮治理，则必须先明确其目标状态，再决定是否补齐为正式工作区。
2. 若某顶层文件名与 OpenClaw 风格相同，但实际承担普通项目文档职责，应标记为例外，不做机械迁移。
3. 若某工作区已有 `.codex/skills/`，但仍同时保留顶层 `skills/`，需要根据其内容判定是否属于 agent 内部资产，再决定迁移或保留。
4. 若执行方发现某些脚本强依赖旧路径，必须先修复依赖，再判定迁移完成，不得以“文件已移动”代替真正生效。

## 8. 依赖项与开放问题
1. 依赖统一的目录映射表和批量搜索替换能力。
2. 依赖执行方能提供 dry-run 与验收报告。
3. 开放问题：`../.codex` 作为共享支撑目录，最终是仅接受目录治理，还是也需要补一个最小 `AGENTS.md` 入口；当前默认按“共享支撑目录”处理，后续可单独升级。
