# 需求概述：跨工作区 OpenClaw 风格能力与目录治理

## 1. 项目背景与问题定义
当前 `../` 下共有 9 个一级目录：`../.codex`、`../agent_gen`、`../Analyst`、`../Analyst2`、`../openclaw`、`../PortraitRpDemo`、`../skill_manager`、`../trainer`、`../workflow`。

现状问题如下：

1. 这 9 个目录的 agent 工作区形态不一致。
2. 其中 8 个目录当前已有 `AGENTS.md`，但只有 `../openclaw` 同时具备较完整的记忆链路和 OpenClaw 风格的 agent 工作文档。
3. `../openclaw` 当前虽然能力较完整，但顶层仍暴露 `SOUL.md`、`MEMORY.md`、`USER.md`、`TOOLS.md`、`HEARTBEAT.md`、`BOOTSTRAP.md` 等 agent 工作文档，不满足“顶层仅保留 `AGENTS.md`”的新治理目标。
4. 其他工作区普遍缺少统一的长期记忆、每日记忆、启动读取顺序、心跳维护和 agent 内部文档落位规则，导致跨工作区切换时恢复成本高、能力不一致。

用户的根本目标不是简单复制 `../openclaw` 的当前目录外观，而是让各工作区具备一致的记忆能力与 OpenClaw 风格的 agent 工作能力，同时把 agent 工作文档和相关资产收纳到 `.codex/` 下，降低顶层噪音与维护成本。

## 2. 目标用户与核心场景
### 目标用户
1. 多工作区 owner：需要在多个 agent 工作区间切换，并希望每个工作区都能保持可恢复、可记忆、可维护。
2. 各工作区内的 agent：需要在每次启动时按统一规则读取身份、用户上下文、短期记忆和长期记忆。
3. 后续执行改造的实现方：需要一套明确的目录契约、迁移映射和验收标准，避免只移动文件而漏掉行为链路。

### 核心场景
1. 新会话启动时，agent 能自动从 `.codex/` 恢复身份、用户上下文和记忆。
2. 日常工作中，agent 能把每日记录写入 `.codex/memory/`，并在主会话下维护 `.codex/MEMORY.md`。
3. 多工作区之间采用一致的 OpenClaw 风格工作区能力，不再出现“有的工作区有记忆，有的没有”的割裂。
4. 工作区根目录保持简洁，agent 工作文档与相关资产集中收纳到 `.codex/`。

## 3. 业务目标与成功指标
### 业务目标
1. 让 9 个一级目录全部进入明确的目标状态，不再存在目录契约模糊区。
2. 让所有 agent 工作区具备统一的记忆能力与 OpenClaw 风格基础能力。
3. 让工作区顶层只保留 `AGENTS.md` 这一份 agent 工作文档，其余 agent 工作文档迁入 `.codex/`。
4. 让迁移不仅完成文件移动，还同步完成 `AGENTS.md`、脚本、技能说明和相关路径引用修复。

### 成功指标
1. `../` 下 9 个一级目录全部完成目标状态判定与迁移计划，不留未归类目录。
2. 8 个已有 `AGENTS.md` 的目录全部支持统一的 `.codex/` 记忆链路和 OpenClaw 风格启动读取规则。
3. 8 个 agent 工作区顶层不再保留 `SOUL.md`、`MEMORY.md`、`USER.md`、`TOOLS.md`、`HEARTBEAT.md`、`IDENTITY.md`、`BOOTSTRAP.md` 等 agent 工作文档。
4. 相关路径引用修复后，已纳入改造范围的 `AGENTS.md`、脚本、技能说明和治理文档中不再引用旧的顶层 agent 工作文档路径。
5. 每个 agent 工作区至少提供 1 组“启动读取成功”和 1 组“记忆写入成功”的验收证据。

## 4. 范围界定
### In Scope
1. `../` 下 9 个一级目录的工作区形态盘点与目标状态定义：
   - `../.codex`
   - `../agent_gen`
   - `../Analyst`
   - `../Analyst2`
   - `../openclaw`
   - `../PortraitRpDemo`
   - `../skill_manager`
   - `../trainer`
   - `../workflow`
2. 8 个已有 `AGENTS.md` 的目录的 OpenClaw 风格能力补齐，包括：
   - 身份与用户上下文文档；
   - 每日记忆与长期记忆；
   - 心跳维护；
   - 工具说明与本地技能入口；
   - 启动读取顺序与安全边界。
3. agent 工作文档与相关工作资产的目录治理，包括迁入 `.codex/`、路径修复和验收。
4. 对 `../.codex` 这种共享支撑目录给出明确目标状态说明，避免既不算工作区、又不受治理。

### Out of Scope
1. 各工作区内部业务代码、页面、接口或运行逻辑的实现改造。
2. 非 agent 工作文档的全面清理或统一重构，例如普通 `README.md`、产品业务说明等。
3. 第三方平台接入、外部通知、聊天渠道接入等产品能力实现。
4. 对各工作区中已有业务数据的内容级清洗。

## 5. 优先级与阶段规划
### P0：盘点与契约确认
1. 盘点 9 个一级目录的当前状态。
2. 明确“哪些目录是 agent 工作区，哪些目录是共享支撑目录”。
3. 明确 `.codex/` 下的标准结构和迁移映射。

### P1：能力基线补齐
1. 为所有 agent 工作区补齐 OpenClaw 风格基础能力。
2. 统一 `AGENTS.md` 启动读取顺序和记忆写入规则。
3. 补齐缺失的 `.codex/skills/`、`.codex/memory/` 及核心 agent 工作文档。

### P2：目录治理与路径迁移
1. 将顶层 agent 工作文档迁入 `.codex/`。
2. 修复 `AGENTS.md`、脚本、技能说明、治理文档中的旧路径引用。
3. 对已有 `memory/`、`skills/`、`tasks/` 等 agent 工作资产进行归类与迁移。

### P3：验收与防漂移
1. 对 9 个目录逐个输出改造结果与例外说明。
2. 提供启动验证、记忆验证、路径验证和回归检查证据。
3. 为后续新建工作区沉淀同一套默认模板与验收门禁。

## 6. 风险、依赖与待确认事项
### 风险
1. 部分工作区当前没有 `.codex/` 目录，也没有记忆链路，若只移动文档而不补启动规则，会形成“文件在，但能力没接通”的伪完成。
2. 现有脚本、技能说明和历史文档中可能硬编码了旧路径；若漏修，会导致启动失败或记忆失效。
3. `../openclaw` 当前的能力最完整，但其目录现状本身与新目标冲突，不能直接照抄外观。

### 依赖
1. 需要一套跨工作区统一的 `.codex/` 目录模板。
2. 需要执行方提供批量迁移与路径修复能力，至少支持 dry-run、映射清单和回滚说明。
3. 需要每个工作区的 `AGENTS.md` 都能被安全更新到同一套读取口径。

### 待确认事项
1. 推断/假设（置信度: 中）：`../.codex` 更适合作为共享支撑目录，而不是独立的 agent 工作区；本轮仍将其纳入治理与目标状态定义，但默认不要求它具备与 8 个 agent 工作区完全相同的记忆装载规则。
2. 推断/假设（置信度: 中）：`memory/`、`skills/`、`tasks/` 等目录若属于 agent 内部工作资产，应统一迁入 `.codex/`；若后续只想迁移文档、不迁目录，需要单独降级范围。

## 8. 能力扩展评估
1. 结论：暂不新增长期 agent 角色。
2. 理由：本轮核心问题是目录契约与能力模板统一，不是职责分工不足。
3. 建议：执行侧可新增一次性的批量迁移脚本或局部 skill，用于目录扫描、路径替换、dry-run 和验收报告生成。
4. 代价：需要维护路径映射与回滚说明，但这类成本低于手工逐目录迁移的返工成本。
